# Scanner detection benchmark

Measures how well a security scanner does on this repo against the **structured
ground truth** in [`reports/vulnerabilities.json`](../reports/vulnerabilities.json)
(generated by `reports/build_report.py`). It scores two things:

- **Detection coverage** — which *positives* a scanner detects, across **all**
  fixtures (not just its primary family): the `vulnerable.*` fixtures, the
  SCA/CI/pipeline cases, and every per-rule example under
  `static/CWE-798/catalogue/` (1k+ secret samples). Reported as a fair
  expected-scope recall plus per-category coverage.
- **Errors** — *false positives* on the *negatives* (`patched.*` remediations and
  the benign look-alikes under `static/CWE-798/false-positives/`), plus *noise*
  (findings on repo files that aren't a known vulnerability).

`run.py` loads the ground truth, obtains scanner findings (by running the
[`security_mcp`](../../security_mcp) CLI or by parsing SARIF you already have),
resolves each SARIF result to a fixture file, scores it, and writes a
coverage + error report.

> The `dynamic/` DAST cases are graded separately (their findings reference URLs,
> not files): match endpoints against `dynamic/openapi.yml` (`x-cwe`). They are
> counted but excluded from the file-based scoring.

## What gets scored

| Ground-truth set | Source in `vulnerabilities.json` | Expectation |
|---|---|---|
| core positives | `findings` (minus DAST) | **should** be detected, ideally with the right CWE |
| secret catalogue | `secret_catalogue` → expands `static/CWE-798/catalogue/**` | **should** be detected as CWE-798 |
| negatives | `false_positives` (`patched.*` + look-alikes) | **should NOT** be reported |

**Every scanner result is matched against the full ground truth**, and a
detection on any fixture is credited regardless of the scanner's family — so a
SAST tool that also catches catalogue secrets gets the credit. The `APPLICABLE`
map in `run.py` only sets each scanner's *expected scope* (the families it is
primarily for), which is used as a fair recall denominator — it **never** decides
which detections count.

## Scoring

For each `(scanner, positive)`:

| status | meaning |
|---|---|
| `✓ cwe`  | reported on that file **and** tagged the **correct CWE** (true positive) |
| `~ hit`  | reported on that file, CWE didn't match / wasn't tagged (true positive) |
| `· miss` | reported nothing on that file (false negative) |

For each `(scanner, negative)`: a report there is a `✗` **false positive**.

Reported per scanner:
- **scope recall** = detected / total over positives in the scanner's *expected
  scope* (the fair denominator)
- **all positives detected** = distinct positives detected across *every* family
  (incl. the 1028-file secret catalogue)
- **coverage per category** = `detected/total` for each of sast / iac / pipeline /
  sca / secret — this is where cross-family detections (e.g. opengrep on catalogue
  secrets) surface
- **false positives** = negatives flagged
- **noise** = findings on repo files that are neither a positive nor a listed
  negative (the DAST app source, graded by endpoint, is excluded)
- **precision** = TP / (TP + false positives + noise)

The secret catalogue is summarized in an aggregate coverage row and the `secret`
category, not expanded into the per-file matrix (which lists only core fixtures).

## Usage

```bash
# 0) (re)generate the ground truth after changing fixtures
python3 reports/build_report.py

# 1) Run scanners via the sibling security_mcp project, then score.
python3 benchmark/run.py --mcp ../security_mcp --tools opengrep,trivy-misconfig

# secret scanners over the whole tree (incl. the CWE-798 catalogue):
python3 benchmark/run.py --mcp ../security_mcp --tools gitleaks,nosey_parker,titus

# CodeQL scans one language at a time:
python3 benchmark/run.py --mcp ../security_mcp --tools codeql --language python

# 2) Score SARIF you already produced (any *.sarif/*.json in the dir).
python3 benchmark/run.py --sarif-dir /path/to/raw

# 3) Just print the ground-truth inventory (no scanners needed).
python3 benchmark/run.py --dry-run
```

Output goes to `benchmark/results/bench_<timestamp>/` (git-ignored):
- `benchmark.md`   — coverage + error tables, catalogue coverage, full matrix
- `benchmark.json` — `{meta, report, ground_truth}` for further processing

## How findings are parsed

`run.py` reads SARIF 2.1.0. For each result it:
- normalizes the artifact URI to a repo-relative path and resolves it to a known
  fixture by basename + suffix (absolute or relative URIs both work);
- extracts CWE tokens (`CWE-\d+`) from the rule id/name/properties/tags and the
  result message — this is how it decides `cwe` vs `hit`.

A scanner that emits no CWE metadata still counts as `hit` (detection) but not
`cwe` (CWE-accurate). That gap is itself a useful signal.

## Relationship to security_mcp

`security_mcp` is the engine (it wraps opengrep, codeql, trivy, osv-scanner,
gitleaks, nosey_parker, titus, plumber, nuclei, …, all emitting SARIF). This
benchmark is the **scoring layer** on top: it points that engine at these
fixtures and grades the output against `reports/vulnerabilities.json`. You can
also bypass the engine and pass any SARIF via `--sarif-dir`.
